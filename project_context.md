## 1. Project Overview

A Python pipeline that generates synthetic tabular data using state-of-the-art generative models (GAN, TVAE), integrates differential privacy (DP) guarantees into the training process, and rigorously benchmarks all generators on both statistical utility and resistance to membership inference attacks (MIA).

**Core problem:** Organisations in healthcare, finance, and government hold valuable sensitive data that cannot be freely shared due to privacy regulations (GDPR, HIPAA). Synthetic data is a privacy-safe stand-in вЂ” fake-but-statistically-realistic records that carry no real individuals.

**Core challenge:** Even synthetic data can leak. A membership inference attack (MIA) probes whether a specific real record was used to train the generator. Differential privacy defends against this by injecting calibrated noise during training so no individual record leaves a detectable fingerprint.

**Deliverables:**
- A modular Python pipeline (data loaders в†’ generators в†’ evaluators в†’ report)
- A benchmark report comparing all generator-epsilon combinations on utility and privacy metrics
- Optionally: a CLI or notebook interface to run end-to-end

## 2. Key Concepts (Plain Language)

### Synthetic Data
Fake-but-realistic records generated by a model that learned the statistical patterns of real data. Looks and behaves like real data but contains no actual individuals.

### Generative Models Used
- **CTGAN** (Conditional Tabular GAN): A GAN where a forger and detective compete вЂ” the forger improves until its fakes are indistinguishable.
- **TVAE** (Tabular Variational Autoencoder): Learns the "recipe" behind the data and generates new samples from it.
- **CopulaGAN**: Models feature correlations using copulas before generation.

### Differential Privacy (DP)
Adds precisely calibrated mathematical noise during training (via DP-SGD) so no single record leaves a detectable fingerprint. Controlled by **epsilon (Оµ)** вЂ” lower Оµ means stronger privacy but lower data quality.

### Membership Inference Attack (MIA)
An attacker is given the synthetic dataset and tries to determine which records from a mixed pool (real train + real test) were used in training. If they can, privacy is compromised.

### Privacy-Utility Trade-off
More privacy (lower Оµ) generally means less useful synthetic data. The benchmark quantifies this trade-off per generator.

## 4. Pipeline Architecture (4 Layers)

### Layer 1 вЂ” Data Ingestion & Preprocessing
- Each dataset has its own loader handling its quirks
- Output: `train.csv`, `test.csv`, `meta.json` (column schema with types, target, cardinalities)
- **Critical:** test set is held out completely вЂ” never seen by generators, used only by the MIA evaluator to simulate attacker knowledge

### Layer 2 вЂ” Synthetic Data Generation
- 6 generators per dataset: CTGAN, TVAE, CopulaGAN (no DP) + DP variants of each
- DP variants run at **Оµ = 1, 5, 10** to produce a trade-off curve
- Each run saves a synthetic CSV labelled with generator name and epsilon value

### Layer 3 вЂ” Evaluation
Two independent evaluators run on every synthetic dataset:

**Utility Evaluator:**
- Column-wise statistics (means, distributions)
- Pairwise feature correlations
- **TSTR** (Train-on-Synthetic, Test-on-Real): XGBoost trained on synthetic, tested on real held-out test set

**Privacy Evaluator:**
- **MIA** (Membership Inference Attack): attacker tries to distinguish real train vs. real test records
- **DCR** (Distance to Closest Record): geometric closeness of synthetic to real records
- **NNDR** (Nearest Neighbour Distance Ratio): relative proximity measure

### Layer 4 вЂ” Benchmark Report
- Aggregates all results into a master comparison matrix
- Rows: generator-epsilon combinations; Columns: utility + privacy metrics across datasets
- Highlights the Pareto frontier: best utility per privacy budget

## 6. Implementation Order

| Stage | What to build | Depends on | Checkpoint |
|---|---|---|---|
| 1 | `config.yaml` + `run_pipeline.py` skeleton | Nothing | Config loads without errors; skeleton runs with placeholder calls |
| 2 | Data loaders + preprocessor (Layer 1) | Stage 1 | All 3 datasets produce clean train/test/meta; test sets are isolated |
| 3 | Non-DP generators вЂ” CTGAN, TVAE, CopulaGAN (Layer 2) | Stage 2 | Each produces a plausible synthetic CSV; categories valid, ranges sensible |
| 4 | Evaluators вЂ” utility then privacy (Layer 3) | Stage 3 | Baseline scores computed for all 3 non-DP generators on all 3 datasets |
| 5 | DP generators at Оµ = 1, 5, 10 (Layer 2 second half) | Stage 3 + 4 | Utility drops as Оµ decreases; MIA resistance improves вЂ” curve is monotonic |
| 6 | Reporting вЂ” aggregator + visualiser (Layer 4) | Stage 4 + 5 | Full benchmark table populated; `run_pipeline.py` runs end to end |

**Golden rule:** Never move to the next stage until the current stage has a passing checkpoint. A broken foundation compounds into undebuggable problems by Stage 5.

## 7. Key Design Principles

- **`config.yaml` is the single source of truth.** No hardcoded paths, epsilons, seeds, or hyperparameters anywhere in source files.
- **Abstract base classes enforce contracts.** Every loader implements `load()` and `split()`. Every generator implements `fit()` and `sample()`. The evaluation layer is agnostic to which generator produced the data.
- **`dp_wrapper.py` is the only place DP logic lives.** It wraps any generator and injects DP-SGD at a specified epsilon. This avoids duplication and makes adding new generators trivial.
- **Test set is sacred.** It never touches the generator training process. It exists solely to simulate the attacker's external knowledge in MIA evaluation.
- **Notebooks are for exploration only.** No pipeline logic lives in notebooks. Anything that works in a notebook gets refactored into `src/` before it counts.
- **`run_pipeline.py` must reproduce everything from scratch.** One command, raw data to benchmark report.
